# Cloudflare Worker Primary Notion API Channel

**Date:** 2026-07-15
**Status:** Approved for implementation planning

## Background

The production blog runs as a single Next.js Docker container on a Tencent
Cloud VPS with Redis. Production logs show intermittent `ETIMEDOUT` and
`ECONNREFUSED` failures while connecting from the VPS to
`https://www.notion.so/api/v3`. A successful uncached request can still take
roughly 1.3 to 3.8 seconds.

The application already protects readers from most upstream failures:

- Public content uses a 300-second ISR eligibility interval.
- Successful data is stored in the normal cache and in a seven-day Redis
  fallback.
- Failed or empty fetches do not replace the last usable value.
- Cache failures fall back from Redis fallback to local stale data before an
  error is exposed.

The transport path is the remaining weak link. The selected change inserts a
small Cloudflare Worker between the Docker application and Notion for the
unofficial `/api/v3` content API. The Worker is a transport proxy only; it does
not become a second application cache.

## Goals

1. Use a Cloudflare Worker as the primary transport for
   `www.notion.so/api/v3` requests.
2. Fall back automatically to direct `www.notion.so/api/v3` access when the
   Worker channel is unavailable.
3. Keep the existing five-minute ISR, Redis, seven-day fallback, stale cache,
   empty-data rejection, and successful-write behavior unchanged.
4. Avoid Mihomo and consume no third-party proxy quota.
5. Prevent a Worker incident from repeatedly adding latency to every Notion
   retry.
6. Restrict the Worker so it cannot be used as a public general-purpose proxy.
7. Record enough channel and latency metadata to verify production behavior
   without logging Notion credentials or content.

## Non-goals

- Moving the website, Next.js runtime, Redis, or Docker deployment to
  Cloudflare.
- Putting the public website hostname behind Cloudflare.
- Caching Notion POST responses in Cloudflare.
- Changing the 300-second ISR interval or the seven-day Redis fallback TTL.
- Refactoring the existing application-level retry counts in this change.
- Proxying Notion images, signed file URLs, video, or other large assets.
- Proxying the separate official `api.notion.com/v1` integration calls used by
  membership, OAuth, payment, or administrative features.
- Adding scheduled refreshes, webhooks, queues, KV, D1, Durable Objects, or
  other Cloudflare storage.

## Options Considered

### Option A: Route Docker through Mihomo

This produced a promising preliminary latency sample, but it consumes a
separate traffic quota, depends on the selected proxy node, and requires
container-to-host proxy plumbing.

**Decision:** Rejected for the requested end state.

### Option B: Set `API_BASE_URL` to a Worker without application fallback

This is the smallest code change, but it replaces one single point of failure
with another. A Worker deployment, credential, DNS, or Cloudflare routing
problem would make the Worker the only content source.

**Decision:** Rejected.

### Option C: Worker primary, direct Notion fallback, existing cache unchanged

The application owns two Notion clients. It uses the Worker normally and opens
a short in-memory circuit after a confirmed Worker-channel failure. While the
circuit is open, requests go directly to Notion. Existing callers, retries,
ISR, and cache behavior remain unchanged.

**Decision:** Selected.

## Architecture

### Cloudflare Worker

The Worker lives under `cloudflare/notion-api-proxy/` and has one responsibility:
forward authenticated server-to-server POST requests from a configured Worker
URL to the same path under `https://www.notion.so`.

It accepts:

- `GET /health` without credentials, returning a small readiness response.
- `POST /api/v3/*` only when `x-notion-proxy-token` exactly matches the Worker
  secret binding `NOTION_PROXY_TOKEN`.

It rejects every other method or path. Before forwarding, it removes the proxy
credential and Cloudflare-specific inbound headers. It preserves the Notion
request body and the Notion authentication headers generated by
`notion-client`.

The Worker never calls the Cache API. Every proxied response carries
`Cache-Control: no-store` and `x-notion-proxy-upstream: notion`. If the Worker
cannot perform its subrequest, it returns HTTP 502 with
`x-notion-proxy-channel-error: 1`. It does not include exception text, request
content, cookies, or tokens in the response or logs.

### Application transport

`lib/db/notion/getNotionAPI.js` owns two lazily created `NotionAPI` clients:

- Primary client: `NOTION_API_PROXY_URL`, with
  `x-notion-proxy-token: NOTION_API_PROXY_TOKEN` and a bounded request timeout.
- Direct client: `https://www.notion.so/api/v3`, without the proxy credential.

If `NOTION_API_PROXY_URL` and `NOTION_API_PROXY_TOKEN` are absent, the module
uses direct Notion exactly as it does today. This is the configuration rollback
path and keeps builds outside production usable.

All exported methods continue to call the same `callNotion(methodName, ...args)`
boundary. Existing call sites do not know which transport succeeded.

### Channel failure classification

The application switches to direct Notion only when the Worker channel itself
failed:

- No HTTP response because of DNS, TCP, TLS, abort-by-timeout, or disconnect.
- A Cloudflare/Worker 5xx response that does not carry
  `x-notion-proxy-upstream: notion`.
- A Worker response carrying `x-notion-proxy-channel-error: 1`.
- Worker authentication or route rejection, which indicates broken proxy
  configuration.

A Notion response forwarded through the Worker carries
`x-notion-proxy-upstream: notion`. Its 400, 401, 403, 404, and 429 statuses are
not retried immediately through the direct channel. Direct access would send
the same invalid or rate-limited request and can worsen rate limiting. Existing
application retry behavior remains responsible for upstream errors.

### Circuit breaker

After a Worker-channel failure, an in-memory circuit opens for 60 seconds.
During that period new calls use the direct client immediately. This avoids
adding a Worker timeout to every nested application retry. A single application
instance is deployed, so a local circuit is sufficient and does not require
Redis coordination.

When the interval expires, the next request probes the Worker. A successful
Worker call closes the circuit. A failed probe reopens it for another 60
seconds.

The circuit does not change the existing application retry loops. In
particular, `withRetry({ retries: 3 })` currently means an initial attempt plus
three retries, while the article block loader has its own retry loop. Those
semantics are intentionally out of scope for this transport change.

## Data Flow

### Normal cache hit

1. A visitor requests a page.
2. ISR/application cache returns usable content.
3. No Worker or Notion request occurs.

### Refresh through Worker

1. Existing application logic decides that Notion data must be fetched.
2. `callNotion` invokes the primary Worker-backed client.
3. The Worker validates the secret, forwards the POST to Notion, and streams
   the response.
4. The application formats the result and writes the existing normal and Redis
   fallback caches.

### Worker failure with direct recovery

1. The Worker request fails with a channel-classified error.
2. The application opens the circuit for 60 seconds.
3. The same logical method call is attempted once through the direct client.
4. A successful direct response is returned to the existing caller and cached
   normally.
5. Further calls during the open interval skip the Worker.

### Both channels fail

1. The direct call throws to the existing retry/cache layer.
2. Existing application retries continue with their current timing and count.
3. If source fetching remains unavailable, `cache_manager` returns the
   seven-day Redis fallback, then local stale data.
4. Failed or empty data never replaces the last successful cache value.

## Configuration and Secrets

Application runtime variables:

```text
NOTION_API_PROXY_URL=https://<worker-host>/api/v3
NOTION_API_PROXY_TOKEN=<random high-entropy secret>
NOTION_API_PROXY_TIMEOUT_MS=6000
NOTION_API_PROXY_CIRCUIT_MS=60000
```

The same token is stored in Cloudflare only as the encrypted Worker secret
`NOTION_PROXY_TOKEN`. It is not committed to Git, embedded in the Docker image,
passed as a Docker build argument, returned by `/health`, or printed in logs.

The Cloudflare account API token used for deployment is an operator credential,
not an application runtime secret. It is supplied only to Wrangler at deploy
time and is never written to the repository or VPS environment.

## Compatibility Notes

The installed `notion-client` version is 7.10.0 and accepts request headers and
timeouts through `ofetchOptions`. The current `kyOptions` property is a legacy
shape and must be replaced without dropping the existing
`syncRecordValues` to `syncRecordValuesMain` URL compatibility behavior.

Only the unofficial content API controlled by `apiBaseUrl` is proxied. Existing
hard-coded `https://api.notion.com/v1` calls and signed asset URLs remain direct,
which avoids changing unrelated product features or sending large media through
the Worker.

## Testing

### Worker unit tests

Using a supplied fake `fetch` implementation, verify:

- Health succeeds without revealing configuration.
- Missing or invalid secret returns 401 and performs no upstream request.
- Non-POST and non-`/api/v3/*` requests are rejected.
- A valid POST preserves path, query, body, and Notion headers while removing
  the proxy secret.
- Proxied responses are marked as Notion upstream and `no-store`.
- An upstream network exception produces the channel-error 502 contract.

### Application unit tests

Using fake Notion client factories, verify:

- Direct-only behavior when proxy configuration is absent.
- Worker success does not call the direct client.
- Worker network/channel failure calls direct once and opens the circuit.
- Calls during an open circuit skip Worker.
- Worker is probed again after 60 seconds and success closes the circuit.
- Forwarded Notion 400/401/403/404/429 responses do not trigger direct fallback.
- Proxy headers are never attached to the direct Notion client.
- Concurrent identical calls retain the existing in-flight deduplication.

### Real integration verification

After deploying the Worker:

1. Send a real `loadPageChunk` fixture through the Worker and compare its
   status and response structure with a direct Notion response.
2. Send the real database `getPage` flow, including `queryCollection`, through
   the Worker.
3. Run at least 200 sequential production-like requests from the VPS and record
   success count and latency percentiles without recording response content.
4. Temporarily configure an invalid Worker host and confirm direct fallback
   succeeds.
5. Temporarily block both transports in a controlled test and confirm the
   existing Redis fallback returns usable data.
6. Restore valid configuration, restart the app, and confirm health, home,
   article, archive, and knowledge-graph smoke tests.

## Deployment and Rollback

Deployment order prevents a partially configured outage:

1. Deploy and test the Worker with its secret.
2. Deploy application code that supports both clients but leave proxy variables
   unset; confirm direct mode.
3. Add the Worker URL/token to the VPS `.env.production` and restart only the
   application container.
4. Run real VPS and public-site verification.

Rollback requires removing `NOTION_API_PROXY_URL` and
`NOTION_API_PROXY_TOKEN` from the VPS runtime environment and restarting the
application container. Redis data and Docker volumes are not deleted, and no
code rollback is required to restore direct mode.

## Success Criteria

- At least 200 VPS-to-Worker production-like Notion requests complete without a
  Worker-channel failure.
- Worker P50/P95 latency and direct P50/P95 are recorded separately.
- Normal application logs identify `worker`, `direct-fallback`, or `direct`
  transport without exposing secrets or content.
- An intentionally unavailable Worker produces a successful direct fallback.
- An intentionally unavailable source path still returns the existing Redis or
  stale value and never caches empty data.
- Public website smoke tests pass after configuration.
- Existing cache, ISR, knowledge-graph, payment, comment, OAuth, and official
  Notion API tests remain green.
