# 主题生态长期运营模型

这份文档用于把“主题共创 + AI 辅助开发”变成长期活动，而不是依赖主理人持续手动推进的一次性活动。

核心原则：**主理人只做方向判断与最终守门，日常运转交给模板、轮值、任务池和社区自组织。**

## 目标

- 让社区持续收集主题想法、拆分任务、提交 PR。
- 让新贡献者有低门槛入口，不需要先理解完整项目。
- 让维护者不用反复解释流程，只需要贴链接和套模板。
- 让主理人从“每个问题都要亲自回答”转为“只处理少数高影响决策”。

## 运行原则

1. **默认公开讨论**：主题想法、问题复现、任务拆分优先放在 GitHub Discussions / Issues，减少私聊沉没成本。
2. **默认模板化**：重复出现的问题沉淀为模板、任务池、提示词或文档。
3. **默认小任务**：新贡献者从文档、截图、单组件、单页面开始。
4. **默认 Draft PR**：不要求一次做完，先开草稿 PR，让社区异步接力。
5. **默认社区互助**：维护者只处理阻塞点，不承诺即时答疑。

## 角色分工

| 角色 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| 主理人 / Founder | 项目方向、架构边界、高风险 PR 最终判断 | 日常答疑、每个主题想法的细节拆分 |
| 主题协调员 | 每月整理主题想法、标记任务、维护任务池 | 承诺亲自实现所有主题 |
| 主题维护者 | Review 指定主题的 PR、补充主题文档 | 处理所有主题的所有问题 |
| 文档维护者 | 把高频问答沉淀到文档 | 私聊逐个解释 |
| 新贡献者 | 认领小任务、开 Draft PR、补截图/文档/组件 | 一次性完成完整复杂主题 |
| 站长用户 | 提供参考站点、截图、使用场景、复现反馈 | 写完整技术方案 |

## 主理人最低负担职责

主理人只需要保留以下动作：

1. 每月看一次“主题方向汇总”。
2. 对高影响方向给出“可做 / 暂缓 / 不适合主仓库”的判断。
3. 只 Review 高风险 PR：
   - 数据层变化
   - 全局配置变化
   - 影响多个主题的公共组件
   - 依赖升级或构建链路变化
4. 指定或确认主题协调员、主题维护者。
5. 对争议做最终裁决。

主理人不需要逐条回复：

- 单个站长的个性化样式问题
- 已有文档覆盖的部署问题
- 还没有参考资料的泛泛主题想法
- 没有复现信息的 Bug

## 月度节奏

建议固定为每月一个轻量循环：

| 时间 | 社区动作 | 主理人动作 |
| --- | --- | --- |
| 第 1 周 | 收集主题想法和参考站点 | 可不参与 |
| 第 2 周 | 主题协调员整理 2 到 3 个候选方向 | 快速判断优先级 |
| 第 3 周 | 社区拆成 `good first issue` / `help wanted` | 可不参与 |
| 第 4 周 | Draft PR、文档、截图、回顾 | 只看需要合并的高影响 PR |

每月只需要发布一条 Discussions 回顾帖：

- 本月新增主题想法
- 本月可认领任务
- 本月已合并 PR
- 下月需要帮助的方向

## 任务流转

```text
微信群 / Telegram / Discussions 中出现主题想法
        ↓
主题风格建议 Discussion
        ↓
主题协调员整理参考站点和适用场景
        ↓
成熟方向转 Theme contribution Issue
        ↓
拆成 good first issue / help wanted
        ↓
新贡献者开 Draft PR
        ↓
主题维护者 Review
        ↓
需要时主理人最终判断
        ↓
合并后更新文档、预览图、回顾帖
```

## 自动分流规则

维护者可以直接复制这些回复，减少重复解释。

### 主题想法还不清楚

```text
这个方向可以先放到 Discussions 的“主题风格建议”模板里。
请补充：参考站点、适用场景、你最想保留的 3 个视觉特征。
成熟后我们再拆成可认领 Issue。
```

### 想用 AI 开发但不知道从哪开始

```text
建议先从 AI 主题开发提示词包开始，不要一次做完整主题。
优先选一个小任务：文章卡片、移动端导航、主题文档或预览图。
可以先开 Draft PR，社区会帮你看方向。
```

### 个性化问题不适合主仓库

```text
这个更像个人站点定制，暂时不适合进入主仓库。
如果你能把它抽象成通用配置项，或者证明多个主题/多个用户都需要，可以再开 Discussions 对齐。
```

### PR 太大

```text
这个 PR 范围偏大，建议拆成几个小 PR：
1. 主题骨架
2. 首页与文章卡片
3. 文章页和列表页
4. 文档、预览图和 manifest
这样更容易 Review，也更容易让社区接力。
```

## 主题协调员职责

主题协调员不是“负责人独自干活”，而是负责让任务流动：

- 每月整理一次主题建议。
- 把成熟想法转成 Issue。
- 给任务加 `good first issue` / `help wanted`。
- 在 Discussions 里贴任务汇总。
- 发现重复问题时补文档或提示词。
- 邀请贡献者开 Draft PR。

## 任务成熟度

| 状态 | 标准 | 下一步 |
| --- | --- | --- |
| 想法 | 只有一句风格描述 | 让提出者补参考站点 |
| 候选 | 有参考站点和适用场景 | 主题协调员整理 |
| 可拆分 | 有基准主题和页面范围 | 创建 Issue |
| 可开发 | 有验收标准和文件范围 | 标记 `help wanted` |
| 新手友好 | 不需要理解全局架构 | 标记 `good first issue` |
| 可合并 | 代码、文档、预览图齐全 | Review / 合并 |

## 衡量指标

不用追求复杂数据，建议每月看 5 个数字：

- 新增主题风格建议数量
- 新增 `good first issue` 数量
- 新增 Draft PR 数量
- 合并的主题/文档 PR 数量
- 由非核心维护者回答的问题数量

最后一个指标很重要：它代表社区是否开始自我闭环。
